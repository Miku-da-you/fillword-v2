# Fillword 重构设计大纲

## 1. 本轮目标

把当前 Fillword 重构成一套可稳定部署在京东云 Linux 服务器上的多人派对 Web 应用，并将多人玩法从“所有玩家填写同一整套字段”升级为“多人模板化 + 2-6 人分配 + 场景化新剧本”。

本轮唯一优先级仍然是：最终稳定运行。

本轮必须达成：

- 清除 Firebase、`localStorage` 演示模式和旧测试页导致的流程分裂
- 保留现有 Socket.IO 实时链路，但将多人逻辑升级为正式的按人数分配模式
- 主持人可在每局建房时自由选择 `2-6` 人
- 每个剧本按人数档位定义字段分配，玩家只填写自己的专属字段
- 新增一批内容更密、梗更多、提示词更丰富的正式剧本
- 按 `/fillword/` 子路径部署到京东云，确保静态资源和 Socket.IO 在反向代理下正常工作

## 2. 产品定位

产品定位为一个轻量、多人、实时、偏搞笑的浏览器派对游戏。

核心体验升级为：

- 主持人选择剧本和本局人数
- 玩家通过房间码快速加入
- 系统按入场顺序给每位玩家分配不同提示词
- 每位玩家只填写自己负责的那一部分内容
- 系统按模板槽位拼装出荒诞、好笑、适合截图传播的结果剧本

视觉气质继续采用“荒诞网感”方向：

- 夸张但可控
- 有互联网梗感，但不做低质土味
- 适合聚会整活和截图发群
- 局部可以疯，整体必须清晰、易用

## 3. 本次重构范围

### 3.1 包含内容

- 前端正式入口重组
- 前端共享脚本与页面脚本模块化
- Socket.IO 联机链路统一
- 服务端房间状态和事件协议重构
- 多人模板化剧本结构设计
- 主持人端人数选择和玩家分配展示
- 玩家端专属字段渲染和提交
- 新剧本扩充与提示词升级
- 结果页展示体验重做
- nginx 子路径部署方案整理
- pm2 进程托管方案整理
- 上线前本地联调与实机联调清单整理

### 3.2 明确清理项

本轮按正式上线版本收口，以下内容直接移除或停用：

- Firebase 相关逻辑
- `localStorage` 演示或降级逻辑
- `dist/` 内的测试页、废弃页、临时调试页
- 任何依赖旧路径或旧事件协议的兼容壳
- “所有玩家填完整一套字段，再合并答案”的旧多人模型

### 3.3 暂不包含

- 登录注册
- 用户账号体系
- 历史战绩持久化
- 排行榜
- 数据库化存档
- 小程序适配
- 主持人中途改人数
- 玩家掉线后的自动重分配
- 自定义剧本编辑器

这些能力只预留扩展位，不作为本轮上线阻塞项。

## 4. 现状问题与重构结论

当前项目虽然已经初步跑通主持人、玩家、Socket.IO 和上线部署，但多人玩法仍存在结构性问题：

1. 旧模型下所有玩家填写相同字段集合，天然会出现“答案覆盖”或“多人贡献感不足”的体验问题。
2. 结果合成逻辑需要额外做补丁式处理，无法从模型层保证每位玩家都真实参与成品生成。
3. 提示词过于基础，多为名词、动词、形容词等抽象类型，玩家难以快速产出更有梗的内容。
4. 主持人无法控制本局人数，剧本也没有按人数档位设计，导致多人体验不可控。
5. 旧页面和旧结构尚未彻底从产品逻辑层收口，后续扩模板会越做越乱。

结论：本轮不再继续修补旧多人模型，而是正式切换到“模板按人数档位定义、字段按玩家分配、结果按槽位拼装”的新结构。

## 5. 目标架构

### 5.1 总体技术栈

- 前端：Vanilla JS + 多页面静态站点
- 后端：Node.js + Express + Socket.IO
- 代理：nginx
- 进程管理：pm2

不引入 React/Vue 等大型前端框架，原因如下：

- 当前交互复杂度不需要重前端框架
- 更利于轻量部署到单台 Linux 服务器
- 手机浏览器访问链路更简单
- 后续维护和快速改版成本更低

### 5.2 前端目录结构

```text
fillword_v2/
  docs/
  public/
    host.html
    player.html
    assets/
    styles/
      tokens.css
      base.css
      host.css
      player.css
      result.css
    scripts/
      shared/
        socket.js
        room-state.js
        templates.js
        utils.js
        ui.js
      pages/
        host.js
        player.js
  server/
    package.json
    server.js
    room-manager.js
    templates.js
    utils.js
```

前端拆分原则：

- HTML 只保留结构骨架和必要容器
- 页面逻辑进入独立页面脚本
- 共享通信、模板、工具、UI 函数拆到共享模块
- 样式按 token、基础层、页面层分离
- 所有资源路径都必须兼容 `/fillword/` 子路径部署

### 5.3 服务端结构

- `server.js`：启动 Express 和 Socket.IO，挂载静态目录，注册路由和事件
- `room-manager.js`：维护房间、玩家、人数档位、字段分配、状态流转、提交和结果生成
- `templates.js`：服务端可用的多人模板定义
- `utils.js`：房间码、校验、错误包装、分配辅助等通用能力

服务端是唯一真相源。前端不能自行判断房间是否存在、谁已提交、谁分到哪些字段、能否生成结果。

## 6. 多人模板化设计

### 6.1 设计原则

多人玩法采用“人数档位化模板”，不采用完全动态平均分配。

原因：

- 结果质量更稳定
- 每个剧本的笑点和节奏可控
- 更容易测试和验收
- 能从数据结构层彻底消灭旧的答案覆盖问题

### 6.2 主持人选人数

主持人建房时必须完成两个选择：

1. 选择剧本
2. 选择本局人数

首版支持范围固定为 `2-6 人`。

规则如下：

- 主持人可以自由选择 `2`、`3`、`4`、`5`、`6` 人
- 剧本卡片必须明确标注自己支持的人数档位
- 如果剧本不支持某人数，则该选项不可创建房间
- 房间创建后目标人数锁定，本局中途不可更改

### 6.3 分配原则

本轮采用“尽量平均”原则：

- 每位玩家分配到的字段数量尽量相同
- 允许最多相差 `1` 个字段
- 如果某个剧本在某人数档位下无法做到自然成文，则该人数档位直接不支持

这里的“平均”服从结果质量，不为了机械平均破坏剧本效果。

### 6.4 分配方式

字段分配不在运行时临时计算语义，而是由模板预先定义好槽位。

运行时行为：

- 玩家加入房间
- 服务端按入场顺序将其绑定到该人数档位中的一个 `assignment`
- 每个 `assignment` 决定该玩家看到哪些提示词
- 玩家提交后，服务端按字段键存储到该玩家名下
- 生成结果时，服务端按模板槽位顺序拼装

这意味着“谁负责什么”是明确、稳定且可测试的，不再依赖模糊的合并策略。

## 7. 剧本数据结构

### 7.1 顶层结构

每个剧本对象至少包含：

```js
{
  id,
  title,
  theme,
  description,
  supportedPlayerCounts,
  variants
}
```

### 7.2 人数档位结构

`variants[count]` 的结构建议为：

```js
{
  promptGroups,
  assignments,
  scriptTemplate,
  resultTitle
}
```

字段含义如下：

- `promptGroups`：本人数档位需要的全部提示词槽位
- `assignments`：每个玩家应负责的字段 key 列表
- `scriptTemplate`：最终结果文本模板
- `resultTitle`：该剧本结果页标题

### 7.3 提示词结构

每个提示词不再只保留简单的 `label` 和 `key`，而是升级为：

```js
{
  key,
  label,
  placeholder,
  category,
  exampleHint,
  tone
}
```

说明如下：

- `label`：字段标题，如“一个非主流网名”
- `placeholder`：输入框占位提示
- `category`：内容分类，供后续筛选或风格控制
- `exampleHint`：示例方向，不提供标准答案
- `tone`：语气标记，如离谱、社死、热血、打工人、土味等

### 7.4 示例

```js
{
  key: "screenName1",
  label: "一个非主流网名",
  placeholder: "例如：冷少殇魂紫月",
  category: "nickname",
  exampleHint: "越像 2012 年空间签名越好",
  tone: "社死"
}
```

## 8. 页面设计

### 8.1 主持人端

主持人端是控场台，用于：

- 选择剧本
- 选择本局人数
- 创建房间
- 查看房间码
- 查看玩家加入情况
- 查看每位玩家被分配的字段数量
- 查看提交进度
- 生成结果
- 关闭房间或开启下一局

页面状态固定为 4 个：

1. 选剧本与人数页
2. 等待玩家加入页
3. 全员提交完成后的控制页
4. 结果展示页

主持人端设计要求：

- 房间码必须大而醒目，可一眼抄写
- 目标人数、当前人数、已提交人数必须一眼可见
- 每个玩家的加入状态、提交状态要清晰展示
- 必要时显示“该玩家负责 `N` 个提示词”
- 按钮文案必须极其明确，不给误点空间

### 8.2 玩家端

玩家端只保留最短路径：

1. 输入房间码和昵称
2. 查看自己被分配的提示词
3. 填写并提交
4. 等待结果
5. 查看结果

玩家端设计要求：

- 手机优先
- 输入框和按钮足够大
- 表单只展示自己负责的字段，不显示其他人的任务
- 每个输入项都有明确、具体、好玩的提示
- 提交后明确进入等待态，不允许重复编辑造成歧义

### 8.3 结果页

结果页是产品记忆点，必须做成“值得截图”的成品页。

表现重点：

- 海报化标题
- 结果剧本文本卡片
- 参与玩家标签
- 每位玩家都真实影响结果内容
- 明显的“再来一局”或“返回首页”操作

## 9. 联机流程设计

### 9.1 角色

- 主持人
- 玩家
- 服务端

### 9.2 核心事件

- `create_room`
- `join_room`
- `submit_answers`
- `leave_room`
- `generate_result`
- `room_state`
- `result_generated`
- `room_closed`
- `error`

### 9.3 房间状态机

房间状态统一由服务端维护，定义如下：

- `waiting`
  主持人已建房，等待玩家加入

- `collecting`
  玩家已加入并处于填写阶段

- `all_submitted`
  所有已分配字段均已提交，等待主持人生成结果

- `result_ready`
  结果已生成，允许所有用户查看

- `closed`
  房间关闭或回合结束

### 9.4 房间真相源

所有房间状态以服务端内存为唯一真相源：

- 房间是否存在
- 主持人是否在线
- 当前剧本和目标人数
- 谁已加入
- 每位玩家被分配了哪些字段
- 谁已提交
- 房间当前状态
- 当前结果

前端只负责渲染和交互，不自行推断核心状态。

### 9.5 关键行为约束

- 主持人创建房间时必须同时提供 `templateId` 和 `targetPlayerCount`
- 只有达到目标人数后，最后一个玩家加入时才完成最终分配
- 玩家加入成功后，服务端返回该玩家专属字段组
- 玩家提交是否完整由前端先做基础校验，服务端再做最终校验
- “是否允许生成结果”只能由服务端判断，前端按钮只是反映状态
- 结果统一由服务端生成并广播，主持人端和玩家端都只展示同一份结果对象
- 不再使用 Firebase 轮询或本地降级逻辑

## 11. 新剧本策略

### 11.1 内容目标

本轮不只修机制，还要同步扩一批“真正好玩”的正式剧本。

新剧本必须满足：

- 提示词更具体，不只停留在词性层
- 结果文本天然带反差和梗
- 多人贡献感强，每个人都能在成品里留下痕迹
- 适合聚会、宿舍、同事局、群聊截图传播

### 11.2 推荐主题池

首批新剧本方向建议覆盖：

- 校园发疯发言
- 社死自我介绍
- 打工人周报
- 乙方汇报现场
- 中二觉醒宣言
- 家族群迷惑文学
- 饭圈式夸夸文
- 土味短视频旁白
- 电竞失败复盘
- 宿舍夜谈怪话

### 11.3 提示词风格

提示词要从“抽象词性”升级为“场景化素材”。

优先使用这类提示词：

- 一个非主流网名
- 一个食物名字
- 一个动漫角色
- 你最讨厌的事情
- 一句很像领导说的话
- 一个听起来就很离谱的部门名
- 一种便利店食物
- 一种适合深夜吃的食物
- 一个小学生作文里会写的梦想
- 一个让人瞬间脚趾抠地的行为

### 11.4 剧本编写原则

- 每句结果文本都要有明确语境，不做纯随机词语堆砌
- 提示词要鼓励玩家产出离谱答案，但不能让人看不懂
- 剧本档位必须从 `2-6` 人视角校验是否自然
- 如果某剧本在某人数档位下质量明显下降，就不支持该档位

## 12. 容错与异常处理

必须做成产品级容错，不能只打印控制台日志。

### 12.1 玩家侧异常

- 房间不存在：提示重新输入房间码
- 房间已关闭：提示本局结束
- 房间人数已满：提示该房间不可继续加入
- 网络断开：显示重连状态和重试入口
- 表单未填完：明确提示缺少内容
- 重复提交：服务端做幂等处理

### 12.2 主持人侧异常

- 玩家中途退出：实时更新人数和提交状态
- 目标人数未满：禁止开始填写或禁止进入生成阶段
- 模板人数档位缺失：阻止建房并提示系统异常
- 主持人断线：进入短时恢复窗口，恢复失败则关闭房间

### 12.3 工程侧异常

- 所有 Socket 事件统一返回 `{ success, error }` 或明确错误对象
- 服务端打印关键事件日志
- 页面出错必须回到安全状态，不允许卡死在半流程
- 资源路径和 Socket 连接地址必须兼容 `/fillword/` 子路径

## 13. 视觉设计方向

### 13.1 主方向

采用“荒诞网感”。

关键词：

- 戏谑
- 反差
- 锋利排版
- 怪但不乱
- 有截图传播欲

### 13.2 视觉原则

- 不使用常见紫白 AI 渐变风
- 强化标题对比和房间码识别度
- 通过局部错位、贴片、标签和强对比色制造性格
- 以可读性优先，不能为了“怪”牺牲可用性
- 提示词表单要比旧版更有引导感，但不能花到影响输入效率

## 14. 移动端要求

主要玩家来自手机浏览器，因此移动端是主平台。

必须满足：

- iPhone 和 Android 主流浏览器可用
- 单手点击方便
- 输入框和按钮尺寸足够大
- 页面切换顺畅
- 长文本正常换行
- 弱网下有明确反馈
- 多个场景化提示词也能在小屏上保持清楚可读

## 15. 部署方案

### 15.1 线上访问路径

线上统一通过子路径访问：

- 主持人端：`/fillword/host.html`
- 玩家端：`/fillword/player.html`

不采用根路径直出，避免未来与同机其他服务冲突。

### 15.2 服务器结构

京东云服务器建议部署为：

- nginx 暴露 80 端口
- Node 应用监听 3000 端口
- nginx 反代 `/fillword/` 和 `/socket.io/`
- pm2 管理 Node 进程

### 15.3 运行稳定性要求

- 进程异常退出可自动拉起
- 重启不因旧进程残留导致 `EADDRINUSE`
- 有基础日志可查
- 更新流程清晰且可重复执行
- 多人房间在 `2-6` 人范围内都能稳定运行

## 16. 实施顺序

### 16.1 第一阶段：清旧链路

- 删除 Firebase 相关逻辑
- 删除 `localStorage` 演示和降级逻辑
- 清理 `dist/` 测试页、废弃页和临时调试入口
- 明确正式部署入口只保留 `public/host.html` 和 `public/player.html`

### 16.2 第二阶段：重构模板结构

- 重写 `templates.js` 为多人模板结构
- 为首批剧本补齐 `2-6` 人档位或合理子集
- 完成提示词元数据升级

### 16.3 第三阶段：重构服务端

- 扩展 `room-manager.js` 以支持 `targetPlayerCount`
- 建立按人数档位的分配模型
- 统一事件协议和错误结构
- 把结果生成逻辑收归服务端

### 16.4 第四阶段：重构前端

- 主持人页增加人数选择

## 17. 成功标准

满足以下条件即视为本轮目标达成：

1. 主持人可稳定通过 `/fillword/host.html` 选择剧本和 `2-6` 人建房。
2. 玩家可通过手机浏览器进入 `/fillword/player.html` 加入同一房间。
3. 不同玩家收到的是不同提示词字段，不再填写同一整套内容。
4. 各人数档位下字段分配尽量平均，最多只相差 `1` 个。
5. 主持人可实时看到加入人数、提交状态和全员是否完成。
6. 主持人可成功生成结果并同步给所有玩家。
7. 所有客户端看到的是同一份结果内容，且成品能体现多人贡献。
8. 房间不存在、房间关闭、人数已满、网络断开、玩家退出、房主掉线等异常都有可理解反馈。
9. 京东云部署方式稳定，可重复启动、更新和查看日志。
