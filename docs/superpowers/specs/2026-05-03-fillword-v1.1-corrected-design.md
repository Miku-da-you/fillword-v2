# Fillword 1.1 修正版设计文档

> **版本目标**：保留 Fillword 1.0 原样，新增真正独立的「海龟汤」和「恐怖怪谈」两个子游戏，三者各有完全不同的玩法规则和交互流程。

---

## 核心理念

**1.0 的错误**：把海龟汤和怪谈做成了"换皮填词"——本质上还是"收集字段 → 阶段化提交 → 系统拼装结果"，这不符合这两种游戏的原始玩法。

**修正方向**：Fillword、海龟汤、怪谈是完全不同的三种游戏，只共享最底层的房间机制（建房、入房、Socket 连接），前端 UI 和服务端逻辑各自独立。

---

## 三个子游戏定义

### A. Fillword（保持 1.0 完全不变）

**玩法**：多人分工填词，主持人一键生成结果。

**流程**：
1. 主持人选剧本、选人数、建房
2. 玩家入房，各自拿到分配给自己的提示词
3. 玩家提交自己的答案
4. 主持人看到全部提交后，点"生成结果"
5. 系统拼接出成品，展示给所有人

**状态**：`waiting → collecting → all_submitted → result_ready → closed`

**结果格式**：一段拼接好的完整文案。

**代码要求**：完全保持 1.0 的 `room-manager.js`、`server.js` 和前端 `host.js/player.js` 不变。

---

### B. 海龟汤（真正的推理问答）

**玩法**：玩家对系统出的题目自由提问，系统根据预置真相回答"是 / 否 / 无关 / 接近了"，玩家在若干轮提问后提交最终真相猜测，系统公布正确答案并评分。

**核心概念**：
- **真相（Truth）**：题目的真实答案，是预置的。
- **问点（AskPoints）**：玩家可能会问的问题类型，比如"受害者的死因是什么"、"凶手用什么凶器"。
- **关键词映射**：每个问点对应多个可接受的提问表述，系统匹配关键词判断回答。
- **提示层级**：每个问点有多个提示阶段，从模糊到清晰，逐步给玩家线索。

**流程**：
1. 主持人选一个题包（Case），开局
2. 系统展示开场题面（opening）
3. **提问阶段**：玩家输入自由问题，系统匹配关键词回答
4. **提示阶段**（可选）：系统根据玩家问不到的点，主动释放提示
5. **猜测阶段**：每位玩家提交自己对真相的猜测
6. **公布真相**：系统展示完整真相故事，公布各玩家猜测与真相的匹配程度

**状态**：`lobby → opening → asking → hinting → guessing → truth_reveal → closed`

**结果格式**：
- 题面
- 各玩家猜测
- 官方真相故事
- 每位玩家的"命中程度"（完全命中 / 部分命中 / 未命中）

**服务端题包结构**：
```js
{
  id: "late-night-elevator",
  title: "深夜电梯",
  opening: "一个人深夜进入电梯，按下楼层后，电梯里响起提示音，他立刻崩溃并大声尖叫。这是为什么？",
  askPoints: [
    {
      key: "scream_cause",
      question: "他为什么尖叫？",
      answerType: "yes_no",         // 系统回答"是"或"否"
      truth: "他看见了镜子里的自己",
      keywordGroups: [
        ["镜子", "镜子里", "倒影"],
        ["自己的脸", "自己的样子", "自己的形象"]
      ],
      hint: "提示层级1：和他看到的东西有关",
      hint2: "提示层级2：不是灵异，是他自己的形象"
    },
    {
      key: "situation",
      question: "他处于什么状态？",
      answerType: "yes_no",
      truth: "他刚哭过，眼眶红肿",
      keywordGroups: [
        ["哭", "流泪", "红肿"],
        ["刚哭", "刚经历", "刚发生"]
      ],
      hint: "提示：和他的情绪状态有关"
    }
  ],
  fullTruth: "他深夜喝完酒回家，进入电梯后，电梯镜子里的他眼眶红肿、满脸泪痕——他自己也不知道什么时候哭成这样，崩溃了。",
  difficulty: 3   // 1-5难度
}
```

**关键词匹配逻辑**：
- 系统把玩家的提问拆词，和每个 `askPoints[keywordGroups]` 做交集匹配
- 有交集 → 回答"是"（如果关键词暗示了正确方向）或"接近了"
- 无交集但问题方向对 → "无关"（但不扣分）
- 关键词完全错误 → "否"

**提示阶段**：每 N 轮玩家没问到关键问点，系统自动释放该问点的 `hint`。

**猜测阶段**：玩家提交一段话（不超过 100 字），系统用关键词相似度判断是否命中真相核心。

### C. 恐怖怪谈（阅读推理）

**玩法**：系统提供一篇规则怪谈/都市怪谈，玩家阅读文本后，根据文本内容选择答案或判断真假，系统根据正确率评分并解锁不同结局。

**核心概念**：
- **故事文本（Story）**：完整怪谈，带有隐藏规则。
- **关键规则（HiddenRules）**：故事背后隐藏的逻辑，玩家需要从文本细节中推断。
- **选择题（Choices）**：每篇故事有 N 道选择题，玩家根据阅读内容选择。
- **结局（Endings）**：根据选择题正确率，解锁不同版本的故事结局。

**流程**：
1. 主持人选一个怪谈包（StoryPack），开局
2. 系统展示故事章节 1
3. **阅读 + 选择**：玩家阅读后，系统给出 2-3 道选择题
4. **判断题**：有 1-2 道真假判断题
5. 系统计算正确率 → 解锁对应结局
6. 展示完整故事 + 结局 + 各玩家得分

**状态**：`lobby → reading → choosing → judging → ending_reveal → closed`

**结果格式**：
- 完整故事文本（带格式）
- 结局类型（真相型 / 开放型 / 反转型）
- 各玩家得分与判断

**怪谈包结构**：
```js
{
  id: "dormitory-rule-13",
  title: "宿舍守则第13条",
  theme: "规则怪谈",
  intro: "新生入学手册里，宿舍守则只有12条。但学长学姐都说，第13条是存在的——只是被藏起来了。",
  chapters: [
    {
      chapterTitle: "第一章：深夜的敲门声",
      content: "凌晨2点，宿舍楼一片寂静。你被一阵轻轻的敲门声惊醒。室友都睡得很沉，敲门声停了。你决定...",
      sceneType: "choice"
    }
  ],
  questions: [
    {
      id: "q1",
      chapterIndex: 0,
      question: "敲门声最可能来自？",
      options: ["隔壁宿舍", "走廊尽头的值班室", "门外不是人"],
      correctIndex: 2,
      hint: "提示：守则里提到过走廊尽头"
    },
    {
      id: "q2",
      type: "true_false",
      question: "判断题：宿舍守则第13条是真实存在的。",
      answer: true,
      hint: "提示：所有"第13条"传说都有个共同点"
    }
  ],
  endings: {
    perfect: "你成功找到了第13条：'不要在凌晨2点后回应任何敲门声。'因为那根本不是敲门——是室友的灵魂在测试你是否还记得规则。",
    partial: "你没有完全找出第13条，但你的判断已经足够让你在这场怪谈中活下来。真相是：第13条只在特定条件下出现。",
    failed: "你不相信第13条的存在，这让你成为了规则怪谈的一部分。醒来时，你发现自己的宿舍已经被重新编号——你现在是13号床位。"
  }
}
```

## 技术架构

### 共享层（最小复用）

只复用：
- `Socket.IO` 连接
- 房间房间码（`roomCode`）机制
- 页面 HTML 外壳和样式
- `pm2` 进程管理

### 服务端结构（三个独立模块）

```
server/
  server.js              ← 共享 Socket.IO + 房间路由
  room-manager.js        ← 只处理 Fillword 房间
  turtle-soup/
    server.js            ← 海龟汤专用逻辑
    cases.js             ← 题包数据
    keyword-matcher.js   ← 关键词匹配引擎
    adjudicator.js       ← 问答判定器
  ghost-story/
    server.js            ← 怪谈专用逻辑
    packs.js             ← 怪谈包数据
    judge.js             ← 判断与结局选择器
```

### 前端结构（三个独立子系统）

```
public/
  host.html              ← 共享主持人外壳，通过 URL 参数区分游戏
  player.html            ← 共享玩家外壳，通过 URL 参数区分游戏
  host-fillword.html     ← Fillword 主持人页（不改动）
  player-fillword.html   ← Fillword 玩家页（不改动）
  host-turtle.html       ← 海龟汤主持人页
  player-turtle.html     ← 海龟汤玩家页
  host-ghost.html        ← 怪谈主持人页
  player-ghost.html      ← 怪谈玩家页
```

**入口策略**：主持人/玩家首页用 URL 参数 `?mode=fillword|turtle|ghost` 跳转到各自独立子系统。不强求合并单页，独立性优先。

---

## 数据与存储

- **服务端内存**：所有状态继续存在服务端内存（`Map` 结构），不引入数据库。
- **题包数据**：海龟汤和怪谈的题包全部写在静态 JS 文件里。
- **无 AI**：所有判定基于关键词匹配和预设逻辑，不调用 AI 接口。
- **房间隔离**：Fillword 房间、海龟汤房间、怪谈房间各自由各自的服务端模块管理，互不干扰。

---

## 测试策略

1. **Fillword 1.0 回归测试**：确保原有 2/3/4 人流程不被破坏。
2. **海龟汤单元测试**：
   - 关键词匹配准确率
   - 各问点的提示层级
   - 猜测判定逻辑
3. **怪谈单元测试**：
   - 选择题对答案判定
   - 真假判断题
   - 结局解锁逻辑
4. **Socket.IO 集成测试**：三套游戏的完整入房 → 交互 → 结果流程。
